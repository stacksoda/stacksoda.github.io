组件的声明周期

组件的生命周期可以被分为三个阶段:

挂载阶段、更新阶段、卸载阶段

挂载阶段

组件被创建,执行初始化,并被挂在到DOM中，完成组件的第一次渲染。依次调用的生命周期方法有：

1.constructor

ES 6 class的构造方法，组件被创建时，首先调用该构造方法，该方法接收一个props参数，props 是从父组件中传入的属性对象，如果父组件中没有传入属性而组件自身定义了默认属性，则这个props指向的就是组件的默认属性。必须在这个方法中首先调用super(props)才能保证props被传入组件中。constructor通常用于初始化组件的state以及绑定处理方法等工作。

2.componentWillMount

该方法在组件被挂载到DOM前调用，且只会调用一次，这个方法在实际项目中很少会用到，因为可以在该方法中执行的工作都可以提前到constructor中。在这个方法中调用this.setState不会引起组件的重新渲染

3.render

定义组件的唯一必要方法，在这个方法中，根据组件的props和state返回一个React元素，用于描述组件的UI，render不负责组件的实际渲染工作，它只是返回一个UI的描述，真正的渲染出页面DOM的工作由React自身负责。render是一个纯函数，在这个方法中不能执行任何有副作用的操作，所以不能执行this.setState

4.componentDidMount

组件被挂载到DOM后调用，且只会调用一次。这时候已经可以获取到DOM结构，因此依赖DOM节点的操作可以放到这个方法中。这个方法通常用于向服务器请求数据。在这个方法中调用 this.setState会引起组件的重新渲染

其中，css-loader 用于使类似@import 和 url(…)的方法实现 require 的功能，style-loader 用于将所有计算后的样式加入页面中

实际运行中当遇到后缀为.css 的文件,webpack先使用css-loader解析这个文件，遇到@import 语句会将对应文件引入并解析，最后计算完的 css 会使用style-loader生成一个内容为最终完成解析的 style 标签,放到 head 标签里

创建新的文件夹，打开，然后执行(该命令不是一定要在空目录下执行)

此时在该新目录下会生成一个 .git的隐藏目录 该目录即仓库

本地仓库由git维护的三部分组成

1.工作目录: 持有实际的文件

2.缓存区:临时保存改动

3.HEAD: 指向最近一次提交后的结果

git文件的状态

查看当前的提交状态：

git库所在的文件夹中的文件大致有4种状态

  1. Untracked: 未跟踪，此文件在文件夹中📂，但并没有加入到git库，不参与版本控制，通过 git add 状态变为 Staged
  2. Unmodify: 未更改，即版本库中文件快照内容与文件夹中完全一致。这种类型的文件有两种去处，如果它被修改，则变为Modified.如果使用 git rm移出版本库，则成为 Untracked文件
  3. Modified: 文件已修改，但并未提交到缓存区，这个文件有两种去处，可通过git add进入暂存Staged状态，使用git checkout即从库中去处文件，覆盖当前修改
  4. Staged: 暂存状态. 执行git commit 则将修改同步到库中，这时库中文件和本地文件又变为一致，文件为Unmodify状态.执行 git reset HEAD filename 取消暂存，文件状态变为 Modified
